# Product

## Overview

Observer is a runtime API security and observability platform that continuously validates live traffic against expected contracts and data access rules.

Its goal is to continously enforce alignment between API design (contract) and real-world behavior (runtime traffic), preventing security risks, data leaks, and unintended functionality.

## Problem

This project primarily mitigates the following OWASP Top 10:2025 risks:

- **A02:2025 Security Misconfiguration** by detecting unintended API behavior
- **A01:2025 Broken Access Control** (via resource-level auditing)
- **A09:2025 Security Logging & Alerting Failures** through structured findings and audit logs

Additionally, it supports malformed input patterns related to **A05:2025 Injection** and promotes secure API design aligned with **A06:2025 Insecure Design**

### Current Pain Points

- What APIs are supposed to do and what they actually do in production often diverge without anyone noticing.
- Developers lack real-time visibility into how APIs handle data, leading to silent failures, data leaks, and inconsistent behavior.
- Security issues caused by misconfigurations and broken access control are typically discovered too late, after deployment or during audits.

## Target Users

- Backend Engineers
- Platform Engineers / Infrastructure Engineers
- Security Engineers / Application Security (AppSec) Teams

## Goals

- Ensure API implementations in production strictly adhere to their defined contracts (OpenAPI), preventing drift and undocumented behavior
- Detect and surface security risks such as unintended data exposure, invalid input handling, and schema violations in real-time
- Provide continuous, runtime visibility into API behavior through structured logging and actionable findings

## Non-Goals

- Replace API gateways, load balancers, or full observability platforms (e.g., not a substitute for Kong, NGINX, or Datadog)
- Perform deep business logic validation or enforce application-specific rules beyond defined contracts and resource permissions
- Act as a primary authentication or authorization system (e.g., not managing identity, roles, or access tokens)

## Core Use Cases

- Detecting API drift by identifying mismatches between live responses and OpenAPI specifications in production environments
- Auditing API responses to ensure sensitive or restricted fields are not exposed based on defined resource permissions
- Monitoring and flagging invalid or unexpected request payloads that could indicate bugs, misuse, or potential security threats

## Success Criteria

- Detect and report at least 95% of contract violations (e.g., schema mismatches, invalid methods, unexpected fields) in observed API traffic
- Introduce less than 5ms average latency overhead per request when operating as a reverse proxy under normal load
- Successfully log and persist 100% of detected findings with structured metadata (request_id, endpoint, violation type) for auditability

## Constraints

- Must operate as a lightweight reverse proxy without significantly impacting request latency or throughput
- Must support standard OpenAPI specifications and JSON-based APIs without requiring changes to existing backend services
- Development timeline is constrained to an MVP scope, prioritizing contract validation and logging over advanced features

## Assumptions

- API contracts (e.g., OpenAPI specifications) are available and accurately reflect intended system behavior
- API traffic consists primarily of JSON-based request and response bodies
- The system can be deployed in an environment where routing traffic through a proxy layer is feasible
