# Architecture

## Overview

This document describes the architecture of Observer, including its major components, data flow, and design decisions.

## High-Level Design

[Brief description of the system shape.]
Observer is designed as reverse-proxy based runtime auditing system that sits between clients and upstream APIs.Its purpose is to inspect live HTTP traffic, validate it against API and resource contracts, and generate structured findings without requiring any changes to upstream applications.

## Major Components

### Proxy Layer

**Responsibility**

- Receives inbound HTTP traffic for configured hosts and forwards requests to appropriate upstream service

- Forwards requests and responses while preserving normal HTTP behavior

- Captures request and response data for auditing purposes without blocking the request flow

- Generates audit jobs and submits them to the asynchronous processing queue

**Inputs**

- Inbound HTTP requests from clients

- Host configuration (routing rules, upstream targets, contract references)

**Outputs**

- Proxied HTTP responses returned to the client

- Audit jobs containing request/response metadata submitted to the queue

**Dependencies**

- Routing and host configurations

- Queue layer for asynchronous audit processing

### Queue Layer

**Responsibility**

- Receives audit jobs generated by the proxy layer and places them into an internal queue for asynchronous processing

- Buffers audit jobs so request forwarding can remain decoupled from audit execution

- Makes queued audit jobs available to worker processes for consumption

**Inputs**

- Audit jobs generated by the proxy layer

**Outputs**

- Queued audit jobs available for worker consumption

**Dependencies**

- Worker layer for job consumption and processing

### Worker Layer

**Responsibility**

- Consumes audit jobs from the queue layer for asynchronous processing

- Coordinates execution of auditing tasks and passes results to downstream layers for persistence and logging

**Inputs**

- Queued audit jobs from the queue layer

**Outputs**

- Processed audit jobs

- Audit results forwarded to the auditing and persistence layers

**Dependencies**

- Queue layer for job retrieval

- Auditing layer for validation logic

- Persistence layer for storing findings

### Auditing Layer

**Responsibility**

- Validates request and response data against API contract definitions

- Validates request and response data against resource contract rules and generates findings for violations

**Inputs**

- Processed audit jobs containing request and response metadata, bodies, and host configuration context

**Outputs**

- Structured audit findings

- Validation results indicating contract and resource rule compliance or violation

**Dependencies**

- API contract definitions

- Resource contract definitions

- Worker layer for audit job execution

- Persistence layer for findings storage

### Persistence Layer

**Responsibility**

- Stores host configurations, audit findings, and other durable system state
- Preserves Observer data across restarts for investigation, troubleshooting, and recovery

**Inputs**

- Host configuration data
- Structured audit findings
- System state updates generated by other layers

**Outputs**

- Persisted host configuration records
- Persisted audit finding records
- Stored state available for retrieval by CLI or other tooling

**Dependencies**

- Configuration file for application and host configurations
- SQLite database to store audit findings

## Data Flow

### HTTP Traffic Flow

1. An inbound HTTP request is received from a client
2. The proxy layer identifies the target host configuration and upstream service
3. The proxy layer creates an audit job containing request metadata and submits it to the queue layer
4. The proxy forwards the HTTP request to the upstream service
5. The upstream service processes the request and returns an HTTP response
6. The proxy layer receives the HTTP response from the upstream service
7. The proxy layer creates an audit job containing response metadata and submits it to the queue layer
8. The proxy forwards the HTTP response back to the client

### Audit Processing Flow

1. Audit jobs are received by the queue layer from the proxy layer
2. Audit jobs are buffered in the queue until they are consumed by available workers
3. The worker layer retrieves audit jobs from the queue
4. The worker layer forwards the job to the auditing layer for validation
5. The auditing layer evaluates the job against API contract definitions and resource contract rules
6. If violations are detected, structured findings are generated
7. Findings and relevant metadata are sent to the persistence layer
8. The persistence layer stores findings for later retrieval and analysis

### Proxy Behavior

- The proxy layer operates transparently and does not modify request or response data under normal conditions
- The proxy layer forwards all valid traffic to upstream services and returns responses to clients withoutalteration.
- The proxy layer generates audit jobs asynchronously to avoid blocking request processing
- The proxy layer returns appropriate error responses (e.g., 502, 504) when upstream communication fails

### Data Flow Summary

- Request and response traffic flow synchronously through the proxy layer
- Audit processing occurs asynchronously via the queue and worker layers
- Findings are generated independently of the request-response lifecycle and persisted for later inspection

## Storage Design

### Primary Storage

- File-based configuration (e.g., JSON or YAML) for host and routing definitions

- File-based storage was chosen because host configurations are portable, easy to edit, and can be version-controlled. This approach also makes it simple to import or align configurations with existing proxy systems such as NGINX.

- Stores:
  - Host configurations
  - Upstream target definitions
  - API contract file references
  - Resource contract file references
  - Proxy-related configuration settings

### Secondary Storage / Cache / Queue

- SQLite database for audit findings

- SQLite was chosen because it provides durable, structured storage without requiring external infrastructure. It allows efficient querying of findings for debugging, analysis, and investigation.

- Stores:
  - Audit findings
  - Request and response metadata
  - Request identifiers
  - Severity, code, and message details
  - Historical audit records

- In-memory queue for audit job processing

- Purpose:
  - Buffer audit jobs generated by the proxy layer
  - Decouple request forwarding from audit execution
  - Support asynchronous worker-based processing
  - Reduce latency in the proxy request path

## Communication Patterns

- **Synchronous request/response:**  
Observer uses synchronous HTTP request/response communication in the proxy path. The proxy layer receives inbound client requests, forwards them to the appropriate upstream service, receives the upstream response, and returns that response back to the client in the same request lifecycle.

- **Asynchronous queue/event flow:**  
Observer uses an asynchronous queue-based workflow for auditing. After capturing request and response data, the proxy layer creates audit jobs and submits them to an internal queue. Worker processes consume these jobs independently of the client request lifecycle, execute contract and resource auditing, and forward findings to the persistence layer.

- **Internal module boundaries:**  
Observer separates responsibilities across internal modules to keep the system modular and maintainable. The proxy layer is responsible for traffic forwarding and audit job creation, the queue layer is responsible for buffering jobs, the worker layer is responsible for consuming jobs, the auditing layer is responsible for validation and finding generation, and the persistence layer is responsible for storing findings and durable system state.

## Scaling Strategy

- Observer scales by separating synchronous proxying from asynchronous audit processing. The proxy layer handles live request/response flow, while the queue and worker layers absorb and process audit workloads independently.

- Expected bottlenecks include proxy-path latency under heavy traffic, queue backlog growth when audit production exceeds worker throughput, and database write contention when a large number of findings are generated.

- The worker layer can be scaled independently by increasing the number of worker routines or moving to a more robust external queue model in the future. The persistence layer can also be evolved independently if SQLite becomes a bottleneck for higher write volume.

## Fault Tolerance / Failure Handling

- **Failure scenario:** Upstream service is unreachable or times out.  
  **Expected behavior:** The proxy returns an appropriate HTTP error response such as 502 Bad Gateway or 504 Gateway Timeout.  
  **Recovery strategy:** Continue serving unaffected hosts, log the failure, and allow requests to succeed automatically once the upstream becomes available again.

- **Failure scenario:** Audit processing falls behind and the queue begins to grow.  
  **Expected behavior:** Request forwarding should continue as long as proxy capacity is not exhausted, while audit processing is delayed.  
  **Recovery strategy:** Increase worker capacity, tune queue behavior, or introduce backpressure and queue limits to prevent unbounded memory growth.

- **Failure scenario:** Invalid contract files or malformed configuration are loaded.  
  **Expected behavior:** The system rejects invalid configuration and prevents auditing from running with broken definitions.  
  **Recovery strategy:** Surface clear validation errors to the user and require corrected configuration before enabling affected hosts.

- **Failure scenario:** Persistence layer temporarily fails or becomes unavailable.  
  **Expected behavior:** Findings may fail to persist, but proxy operation should remain available whenever possible.  
  **Recovery strategy:** Log persistence failures, preserve proxy stability, and allow findings storage to resume once the storage layer recovers.

## Security Considerations

- **Auth/authz consideration:**  
  Observer is currently a development tool and may operate without full authentication in local environments, but any management or configuration interfaces exposed beyond local use should require authentication and authorization.

- **Input validation consideration:**  
  All untrusted input must be validated before use, including configuration files, host definitions, API contracts, resource contracts, and observed HTTP payloads used in auditing.

- **Secrets/configuration handling:**  
  Sensitive values such as credentials, tokens, or private upstream configuration should not be hardcoded. Configuration files should avoid storing secrets in plaintext where possible and should support environment-based overrides or secure external secret handling in future deployments.

- **Audit/logging consideration:**  
  Logs and findings should contain enough detail for troubleshooting and investigation, but should avoid unnecessarily exposing sensitive request or response data. Where appropriate, sensitive fields should be masked or omitted from persisted findings.

## Tradeoffs

- Using asynchronous workers reduces proxy-path latency, but it also means findings may be delayed rather than available immediately in the request lifecycle.

- Using file-based configuration improves portability and makes import/export easier, but it introduces the risk of manual misconfiguration and requires clear validation.

- Using SQLite keeps the system simple and easy to run locally, but it may become a limitation under higher write volume or multi-instance deployments.

## Future Architectural Evolution

- Replace the in-memory audit queue with an external queue or broker to support larger workloads, better durability, and more flexible worker scaling.

- Expand the auditing model to support more complex resource relationships, including multi-table operations, derived fields, and aggregated responses.

- Evolve Observer from a local development tool into a more production-capable platform with stronger access controls, richer metrics, and optional distributed deployment.
