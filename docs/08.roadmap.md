# Roadmap

## Overview

This roadmap outlines the planned evolution of **Observer**, a reverse-proxy-based runtime auditing system that inspects HTTP traffic and validates it against API contracts and custom rules contracts.

---

## Current State

- Architecture, requirements, product direction, and data flow have been defined
- Core proxy, queue, worker, auditing, and persistence layers have been established
- Host-based OpenAPI contract loading is implemented
- Request-side API auditing is implemented for the MVP scope
- Audit jobs and findings are persisted for later review
- Current limitation: response-side contract auditing and custom rules contract support are not yet implemented

---

## Near-Term Goals

### Phase 1: Core Runtime Pipeline (MVP)

**Goal**  
Establish a fully working end-to-end pipeline that proxies HTTP traffic, generates audit jobs, processes them asynchronously, and persists basic findings.

**Includes**

- File-based host configuration loader and validation
- Basic host-based reverse proxy with upstream forwarding
- Audit job model:
  - request jobs
  - response jobs
  - failure jobs
- In-memory queue for audit job buffering
- Worker pool for consuming audit jobs
- SQLite persistence for audit jobs and findings
- Request and response capture for auditing
- Basic failure findings for upstream errors and timeout scenarios

---

### Phase 2: API Contract Auditing

**Goal**  
Introduce validation of HTTP traffic against API contract definitions to detect inconsistencies between expected and actual API behavior.

**Includes**

- API contract file loading and validation
- Host-specific API contract binding
- Request-side contract validation:
  - path exists in contract
  - HTTP method is allowed
  - request content type is allowed
  - request body presence/absence is valid
  - request body format matches the declared media type
  - basic JSON schema conformance
- Response-side contract validation:
  - expected status codes
  - response content type
  - response body format
  - basic JSON schema conformance
- Generation of structured findings for contract violations

---

## Mid-Term Goals

### Phase 3: Custom Rules Contract Support

**Goal**  
Extend auditing capabilities beyond OpenAPI by allowing users to define custom runtime auditing rules for organization-specific security, compliance, and API behavior expectations.

**Includes**

- Custom rules contract file loading and validation
- Host-specific custom rules contract binding
- Rule matching by:
  - host
  - path
  - HTTP method
  - request headers
  - response status code
  - response headers
- Request-side custom rule validation:
  - required headers
  - disallowed headers
  - required content types
  - disallowed content types
  - required JSON fields
  - disallowed JSON fields
  - sensitive values in request bodies
- Response-side custom rule validation:
  - required response headers
  - disallowed response headers
  - sensitive fields in response bodies
  - disallowed response fields
  - expected status code patterns
- Integration of custom rule findings into the existing audit pipeline

---

### Phase 4: Observability and Tooling

**Goal**  
Provide visibility into system behavior and findings to support debugging, analysis, and development workflows.

**Includes**

- CLI for listing and inspecting findings
- CLI for listing audit jobs and request history
- Structured logging across:
  - proxy layer
  - queue layer
  - worker layer
  - auditing layer
  - persistence layer
- Request ID tracking across the full pipeline
- Basic metrics:
  - queue depth
  - worker throughput
  - processing latency
  - findings generated
  - dropped audit jobs
- Improved local testing utilities and example contracts

---

## Long-Term Goals

### Phase 5: Scalability and Distributed Processing

**Goal**  
Evolve Observer to handle larger workloads and more complex environments.

**Includes**

- Replace in-memory queue with an external queue or message broker
- Support distributed worker processing
- Improve persistence layer scalability beyond SQLite if needed
- Introduce queue backpressure and workload balancing strategies
- Add configurable audit sampling or rule-specific enablement
- Improve storage retention and cleanup strategies

---

### Phase 6: Advanced Auditing and Platform Expansion

**Goal**  
Transform Observer into a more advanced application security and observability platform.

**Includes**

- Advanced OpenAPI schema validation support:
  - `oneOf`
  - `anyOf`
  - `allOf`
  - `enum`
  - `nullable`
  - `additionalProperties`
  - `minLength`
  - `maxLength`
  - `minimum`
  - `maximum`
  - advanced `format` validation
- More advanced custom rules:
  - authorization expectations
  - sensitive data exposure rules
  - compliance-oriented checks
  - route-specific security policies
  - environment-specific rules
- Advanced analysis of traffic patterns and anomalies
- Optional UI/dashboard for findings visualization
- Enhanced integration with development and CI/CD workflows
- Exportable reports for audit and review

---

## Risks / Dependencies

- Dependency on correct and well-formed API contract definitions
- Dependency on accurate and maintainable custom rules contracts
- Risk of false positives if contracts or custom rules drift from actual intended behavior
- Risk of queue backlog growth under high traffic if worker throughput is insufficient
- SQLite write limitations under high-volume audit scenarios
- Potential performance impact if audit job generation or processing is not properly decoupled
- Complexity of accurately validating real-world API responses against flexible schemas
- Complexity of designing custom rules that are expressive without becoming difficult to maintain

---

## Notes

- Priorities may change based on performance results, usability feedback, and real-world testing scenarios
- Initial focus is on validating architecture and core pipeline before expanding auditing complexity
- OpenAPI contract auditing should remain the foundation for expected API behavior
- Custom rules contracts should extend OpenAPI rather than replace it
- System design favors incremental evolution from a local development tool toward a more scalable platform
