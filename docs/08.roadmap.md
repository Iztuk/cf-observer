# Roadmap

## Overview

This roadmap outlines the planned evolution of **Observer**, a reverse-proxy-based runtime auditing system that inspects HTTP traffic and validates it against API and resource contracts.

---

## Current State

- Architecture, requirements, and data flow have been fully defined
- Initial design for proxy, queue, worker, auditing, and persistence layers is established
- Known limitation: No working runtime pipeline implemented yet (no proxying, queueing, or auditing in execution)

---

## Near-Term Goals

### Phase 1: Core Runtime Pipeline (MVP)

**Goal**  
Establish a fully working end-to-end pipeline that proxies HTTP traffic, generates audit jobs, processes them asynchronously, and persists basic findings.

**Includes**

- File-based host configuration loader and validation
- Basic host-based reverse proxy with upstream forwarding
- Audit job model (request, response, failure)
- In-memory queue for audit job buffering
- Worker pool for consuming audit jobs
- SQLite persistence for audit findings
- Minimal auditing (basic findings such as upstream failures and request/response capture)

---

### Phase 2: API Contract Auditing

**Goal**  
Introduce validation of HTTP traffic against API contract definitions to detect inconsistencies between expected and actual API behavior.

**Includes**

- API contract file loading and validation
- Request-side contract validation (path, method, content type, body rules)
- Response-side contract validation (status codes, content type, schema basics)
- Generation of structured findings for contract violations

---

## Mid-Term Goals

### Phase 3: Resource Contract Enforcement

**Goal**  
Extend auditing capabilities to enforce data-level access rules and detect improper data usage or exposure.

**Includes**

- Resource contract file loading and validation
- Request-side resource validation (write permissions, allowed fields, mutability)
- Response-side resource validation (read permissions, sensitive field exposure)
- Integration of resource-based findings into audit pipeline

---

### Phase 4: Observability and Tooling

**Goal**  
Provide visibility into system behavior and findings to support debugging, analysis, and development workflows.

**Includes**

- CLI for listing and inspecting findings
- Structured logging across proxy, queue, worker, and persistence layers
- Request ID tracking across the full pipeline
- Basic metrics (queue depth, worker throughput, processing latency)

---

## Long-Term Goals

### Phase 5: Scalability and Distributed Processing

**Goal**  
Evolve Observer to handle larger workloads and more complex environments.

**Includes**

- Replace in-memory queue with external queue or message broker
- Support distributed worker processing
- Improve persistence layer scalability beyond SQLite if needed
- Introduce queue backpressure and workload balancing strategies

---

### Phase 6: Advanced Auditing and Platform Expansion

**Goal**  
Transform Observer into a more advanced application security and observability platform.

**Includes**

- Support for multi-resource and relational validation
- Customizable audit rules and policies
- Advanced analysis of traffic patterns and anomalies
- Optional UI/dashboard for findings visualization
- Enhanced integration with development and CI/CD workflows

---

## Risks / Dependencies

- Dependency on correct and well-formed API and resource contract definitions
- Risk of queue backlog growth under high traffic if worker throughput is insufficient
- SQLite write limitations under high-volume audit scenarios
- Potential performance impact if audit job generation or processing is not properly decoupled
- Complexity of accurately validating real-world API responses against flexible schemas

---

## Notes

- Priorities may change based on performance results, usability feedback, and real-world testing scenarios
- Initial focus is on validating architecture and core pipeline before expanding auditing complexity
- System design favors incremental evolution from a local development tool toward a more scalable platform
